# 説明スタイル ルール

このプロジェクトでユーザーに数学概念を説明するときの厳守ルール。

## 言語
- 必ず日本語で応答する
- 専門用語は初出時に「日本語（English）」併記

## 順序（厳守）
1. 直感（一言）
2. 記号の読み方（数式があれば）
3. 具体例（数字を入れた最小例）
4. **Python 実装（標準形式）** — NumPy / SymPy / SciPy
5. **Python 実装（JAX形式）** — `jax.numpy`, `jit`, `grad`, `vmap`
6. 厳密な定義（必要なら）
7. 応用先（ML/AI で何に使うか）

## 各章の md には必ず Jupyter 起動ガイドを入れる

各章 md の冒頭に「**💡 このページのコードを動かすには**」ボックスを置き、以下を必ず含める:

1. `uv run jupyter lab` のコマンド
2. 該当 notebook へのファイルツリー上のパス
3. `start_here/00_pet_terminal/` への誘導（CLI/uv が分からない人向け）
4. `start_here/` および `00_notation/` への誘導（数学の前提知識が不安な人向け）

詳細テンプレートは AGENTS.md 参照。
**置き去りにする学習者を生まない**ことが本リポジトリの大原則。

## 「標準形式」と「JAX形式」の併記ルール（厳守）

すべての計算コードは、原則として標準形式（NumPy など）と JAX 形式の**両方**を提示する。

### 順序
1. **標準形式を先に**書く（数式 → コード対応が直感的）
2. **JAX形式を後に**書く（自動微分・JIT・vmap の恩恵を示す）
3. 両者の**出力が一致する検算**を末尾に置く

### 「JAX形式が標準形式より明らかに優れる」場面で強調
- 微分（手動導出 → `jax.grad` で関数を渡すだけ）
- 同じ計算の繰り返し（`vmap` で1行）
- 高速化（`@jit` を付けるだけ）

### 例外（JAX形式を省略してよい場合）
- 入門・記号解説など、計算が中心ではない説明
- `sympy` による記号計算（JAXは数値計算向け）
- 集合・組合せ列挙など、配列演算でない処理

## 禁止事項
- 「自明」「明らか」「簡単に」という表現を使わない
- 定義だけで終わる回答をしない
- 数式だけ提示して Python を省略しない
- 一度に複数の概念を詰め込まない

## コード規約
- 型ヒント必須（`x: np.ndarray` のように）
- docstring 必須
- 配列の shape はコメントで明記
- 単位（秒、cm、度、確率など）はコメントで明記
- マジックナンバーは定数化

## 確認の入れ方
- 大きな概念を説明し終わったら「ここまで大丈夫ですか？」と確認
- ユーザーが「もう少し詳しく」「別の例を」と言えるよう余地を残す
