# コラム 01: UNIX の哲学 — 小さく作って組み合わせる

> *50年前のエンジニア達が決めた "考え方" が、今のスマホ・AI・ChatGPT の根っこにある話。*

## 1969年、ベル研究所の倉庫で

1969年、アメリカ・ニュージャージー州の **AT&T ベル研究所**。
**ケン・トンプソン** という研究者が、研究所の片隅に放置された PDP-7 という古いコンピュータで、独学で OS を作り始めました。

なぜ?
当時、ベル研究所は MULTICS という壮大な OS プロジェクトに参加していましたが、複雑すぎて頓挫します。
ケン・トンプソンは「**もっとシンプルな OS が作れるはず**」と、こっそり一人で書き始めたのです。

その OS の名前は、最初 **UNICS** (UNiplexed Information and Computing Service)。
後に **UNIX** (ユニックス) と呼ばれるようになります。

開発に **デニス・リッチー** が加わり、彼が UNIX を書くために C 言語まで作ってしまいました。
**UNIX のために言語を作った**のです。

そしてこの2人が確立した**哲学** が、後に世界を変えます。

## UNIX の哲学 — Doug McIlroy の言葉

UNIX 開発に関わった **ダグ・マキルロイ** が、1978年にこう書きました:

> **(i) Make each program do one thing well.**
> (ひとつのことを、うまくやれ)
>
> **(ii) Expect the output of every program to become the input to another.**
> (どのプログラムの出力も、別のプログラムの入力になることを想定せよ)
>
> **(iii) Design and build software early, even operating systems, to be tried by months. Throw away the clumsy parts and rebuild them.**
> (早く設計・実装し、何ヶ月か使ってみる。ダメな部分は捨てて作り直せ)
>
> **(iv) Use tools in preference to unskilled help. Even if you have to detour to build the tools.**
> (人手でやるより、道具を使え。道具を作る回り道をしてでも)

これが**UNIX の哲学**。
50年経った今でも、**プログラミングの最高の指針**として引用され続けます。

## なぜ「小さく作る」のか

普通に考えると「**全部入りで便利な大きなソフト**」のほうが良さそうですよね。
でもUNIX は逆を行きました。**意図的に小さく作る**。

理由は3つ:

### 1. 理解しやすい
1万行のコードより、100行のコードのほうが読める。
**バグも少ない、修正もしやすい**。

### 2. テストしやすい
小さい範囲なら、動作の正しさを保証しやすい。

### 3. 組み合わせると無限の可能性
小さな部品が10個あれば、組み合わせは10×10×10×... と**指数的に増える**。
1個の巨大ソフトでできることは限られるけど、組み合わせは無限。

## 例: 「ファイル一覧から ".py" だけ取り出す」を考える

### Windows の昔の発想
専用ソフトを作る:
- 「Python ファイル抽出ツール v1.0」
- ボタンが3つくらいついた GUI
- そのソフトしかその仕事はできない

### UNIX の発想
小さい道具を組み合わせる:
```bash
ls | grep .py
```

`ls` は「**並べる**」専用、`grep` は「**抽出**」専用。
それだけしかできないけど、組み合わせると上の仕事ができる。
さらに:

```bash
ls | grep .py | wc -l           # Python ファイルの数
ls | grep .py | sort             # アルファベット順に並べる
ls | grep .py | xargs cat        # 全部の中身を表示
```

**4つの道具で、無限のバリエーション**。
これが UNIX の強さです。

## 哲学の現代版

時代が進み、UNIX 哲学はもう少し洗練されて言い直されています。
有名なのが **エリック・レイモンド** が『**UNIX という考え方**』(2003年) で書いた17の原則。

抜粋:
- **明白さの原則**: 巧妙さより明白さ
- **最小驚きの原則**: 最も驚かない動作を選べ
- **沈黙の原則**: 言うことがなければ何も言うな
- **修復の原則**: 失敗したら、騒ぎ立てて、できるだけ早く失敗せよ
- **経済性の原則**: プログラマーの時間は、マシンの時間より高価だ

これらは、現代のソフトウェア開発でも**通用する原則**です。

## 沈黙の原則の例

UNIX のコマンドは、**成功したら何も言わない**のが基本:

```bash
$ cp a.txt b.txt
$                    # 何も表示されない = 成功
```

```bash
$ cp a.txt b.txt
cp: cannot stat 'a.txt': No such file or directory
$                    # 失敗したときだけメッセージ
```

「**便りがないのが良い便り**」というのは、UNIX の作法そのもの。
これに慣れると、**Windows のソフトの "操作完了!" ダイアログ**が、ちょっとうるさく感じます。

## 哲学が現代に与えた影響

UNIX 哲学は、UNIX だけのものじゃありません。
現代のあらゆるところに浸透しています。

### マイクロサービス
**「巨大な1つのアプリ」ではなく、「小さなサービスの集合」**で大きなシステムを作る、という考え方。
Netflix、Uber、Amazon の裏側は、何百ものマイクロサービスの組み合わせ。
**完全に UNIX 哲学**。

### 関数型プログラミング
**「副作用のない小さな関数を組み合わせる」** という、現代の人気スタイル。
JAX も、関数型プログラミングの影響を強く受けています (このリポジトリの 07 章で扱ってますね)。

### Docker / コンテナ
**「1コンテナ = 1つの仕事」** が基本ルール。
これも UNIX 哲学の継承。

### Lego ブロック
これは UNIX より古いけど、考え方は同じ。
**小さな汎用部品を組み合わせて、無限の作品**を作る。

### 生成AI へのプロンプト
AI に「**全部やって**」と言うと、ぼんやりした結果が出ます。
「**まずこれをやって、次にこれをやって、最後にこれを**」と段階的に伝えると、精度が上がる。
これも UNIX 哲学の考え方。

**AI 時代に成功する人は、UNIX 哲学を知ってる人**かもしれません。

## トンプソンとリッチー、その後

UNIX を作った2人は、後にコンピュータ科学界の最高栄誉**チューリング賞** (1983年) を受賞します。
ノーベル賞には情報科学部門がないので、これがコンピュータ界のノーベル賞相当。

**ケン・トンプソン** は晩年、Google に移って Go 言語を共同設計。
**デニス・リッチー** は2011年に亡くなりました (奇しくもスティーブ・ジョブズ死去の1週間後でした)。

ジョブズの死は世界中で報道されましたが、リッチーの死はほとんどニュースにならなかった。
でも、**ジョブズの製品が動いている根っこのコードは、リッチーが書いた**ものだったのです。

## あなたの生活への遺産

今、あなたが触っているもの:
- **Mac** → BSD UNIX 系
- **iPhone** → BSD UNIX 系
- **Android** → Linux カーネル
- **AWS のサーバー** → Linux
- **ChatGPT のサーバー** → Linux
- **Tesla の車載システム** → Linux

**世界の8割以上のコンピュータが、UNIX 系の血を引いています**。
1969年の倉庫で始まったプロジェクトが、**人類の文明そのものを支えている**。

ちょっとロマンを感じませんか?

## まとめ

UNIX の哲学を一行で:

> **小さく、ひとつのことをうまくやる道具を、組み合わせて大きな仕事をする**

そして、これは**現代の AI とプロンプト**にも、そのまま当てはまります。

**1969年のアイデアが、2026年の AI 時代にもそのまま生きている**。
こういうのが、技術のロマンです。
